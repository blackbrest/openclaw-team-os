# OpenClaw Team OS UI 规范

本文件作为 `OpenClaw Team OS` 当前阶段的 UI 设计准则与实现参照。

目标不是写一份泛泛的设计理论，而是把已经确认过的产品判断、视觉约束和布局规则固定下来，作为后续界面修改的默认依据。后续如无特殊说明，优先遵循本规范，不再逐项重新讨论相同细节。

## 1. 产品界面定位

### 1.1 产品气质

- 桌面优先，不做移动优先的轻应用视觉。
- 整体观感应接近「桌面工作台 / 创作操作系统」，而不是营销落地页。
- 视觉语言应克制、稳定、专业，避免花哨、拥挤、廉价感。
- 风格关键词：`dark`, `cinematic`, `operating system`, `quiet`, `premium`, `focused`.

### 1.2 默认审美判断

- 优先“稳”和“整齐”，再追求“炫”。
- 优先“信息层级清楚”，而不是装饰感。
- 优先“留白有节奏”，而不是大面积空洞。
- 优先“克制的高级感”，避免按钮、描边、阴影都过重。

## 2. 全局布局规范

### 2.1 壳层结构

桌面端默认结构：

1. 左侧导航栏 `client-rail`
2. 顶部固定 tab 栏 `workspace-topbar`
3. 主工作区 `workspace-surface`

### 2.2 尺寸基准

当前默认值：

- 顶部 tab 栏高度：`68px`
- 页面大标题头区高度：`120px`
- 页面主内容区从顶部栏下方开始排布
- 左侧导航栏为固定侧栏，不与正文抢垂直节奏

### 2.3 布局原则

- 顶部栏必须始终固定在页面最上层，滚动时不消失。
- 页面主内容必须从顶部开始排布，禁止出现因为容器拉伸导致的无意义上方大空白。
- 大标题头区统一高度，避免不同页面出现不同首屏节奏。
- 主工作区为“单核心内容区”，不要做多块同等级主视觉竞争注意力。

## 3. 顶部栏规范

### 3.1 结构

顶部栏只承担 3 类信息：

1. 当前工作区标题
2. 少量高频全局操作
3. 个人账户入口

### 3.2 已锁定决策

- 去掉无效的 `角色 org_admin`、`单位 none` 展示。
- 去掉无意义的顶部大 CTA，例如“先补第一位 Bot”。
- 用户区只保留圆形头像入口，不在头像旁展示英文身份说明。
- 通知和设置图标使用简约、清晰、线性风格。
- 头像 hover 展开账户菜单。

### 3.3 顶部栏视觉原则

- 顶部栏高度必须稳定，不能因为文案长度或页面切换被撑高。
- 图标区应保持轻量、安静，不做过重发光和夸张 hover。
- 标题与操作区之间优先保证右侧操作可见，左侧文案可省略，不可反过来。

## 4. 页面大标题头区规范

### 4.1 用途

大标题头区只承担：

1. 页面归属提示 `kicker`
2. 页面主标题
3. 一行副标题说明

### 4.2 尺寸与对齐

默认桌面端：

- 头区高度：`120px`
- 文案组整体在头区内垂直居中
- 文案左对齐，但“整个文案组”要位于头区上下边界的视觉中线附近

### 4.3 内容约束

- `kicker` 用于品牌/空间归属，如 `OPENCLAW STUDIO`
- 主标题只保留页面名，如 `招聘`、`员工`、`团队`
- 副标题保持 1 行优先，超出可截断，不允许把头区再次撑高
- 不在头区里塞右侧元信息块
- 不在头区下方重复显示组织切换成功、session 之类提示

### 4.4 视觉原则

- 头区不能过高，避免浪费首屏空间
- 也不能压得过紧，避免标题组显得“卡住”
- 头区的目标状态是：`有呼吸感，但不空`

## 5. 左侧导航栏规范

### 5.1 结构

左侧导航栏包含：

1. 品牌区
2. 主导航分组
3. 次级治理分组
4. 底部账户卡片

### 5.2 视觉原则

- 左栏应像“桌面应用侧栏”，而不是网站菜单。
- 当前激活项要明显，但不能过度发光。
- 文字和图标必须统一垂直节奏。
- 分组标签使用英文大写小号字即可，不要抢戏。

### 5.3 导航项规范

- 激活态优先通过底色、轻描边、轻高亮实现。
- 图标容器不宜过大，避免挤占标签空间。
- 导航项名称优先中文，避免中英混杂造成层级噪音。

## 6. 卡片与内容区规范

### 6.1 Surface Section

`surface-section` 是页面内容的一级分区。

规则：

- 一级 section 默认透明，不做厚重卡片包裹
- 通过细分隔线区分层级
- 一级 section 标题字号应明显小于页面大标题

### 6.2 Recruit Card / Module Card

招聘卡片是当前产品的核心业务卡片，规则如下：

- 优先横向铺开，不要做过高卡片
- 内容层级：分类 strap -> 标题 -> 摘要 -> 能力标签 -> CTA
- 卡片风格应偏“冷静的专业工作单元”，不做电商式高刺激按钮
- 同屏卡片间距优先整齐均匀，不做随机高低错落

### 6.3 Empty State

空状态只给 3 类信息：

1. 当前没有什么
2. 为什么没有
3. 下一步去哪里

禁止：

- 冗长说明
- 重复组织信息
- 多个同级 CTA

## 7. 账户菜单规范

### 7.1 结构

头像 hover 菜单当前采用“深色悬浮卡片”结构：

1. 用户头部
2. 会员区
3. 功能菜单
4. 退出登录

### 7.2 视觉原则

- 悬浮层必须比页面主体更轻、更精致
- 不允许做成巨大、厚重、笨拙的黑色面板
- 圆角、阴影、描边都应克制
- 头像入口与菜单体量必须协调

### 7.3 内容原则

- 头像旁不展示冗余英文身份
- 菜单项名称优先中文
- 占位功能可以保留，但视觉上要像真实菜单，不像临时调试面板

## 8. 字体与排版规范

### 8.1 总体原则

- 中文信息为主，英文只用于少量品牌化 `kicker` 和分组标签
- 英文不承担主要业务信息
- 字体层级必须明确，避免页面每一层都“很大”

### 8.2 层级建议

- 页面大标题：强识别，但只在头区使用
- 一级 section 标题：明显但收敛
- 卡片标题：信息清晰，不能抢过页面级标题
- 说明文字：以可读性优先，避免过淡

### 8.3 字重与间距

- 中文大标题可偏重，但副标题和说明文必须回归正常字重
- 英文 `kicker` 使用大写和拉开字距，但字号必须小
- 不要让字距过大导致“海报感”压过“工作台感”

## 9. 色彩与材质规范

### 9.1 背景

- 默认深色背景
- 使用轻微渐变、细颗粒或低对比纹理增强空间感
- 不允许大面积纯黑死板铺底

### 9.2 表面层级

- 主背景最暗
- 主工作面板略亮
- 浮层再略亮或更清晰
- 强调色只用于焦点，不可满屏到处点亮

### 9.3 强调色

当前可接受的强调方向：

- 紫色系：品牌与标题 kicker
- 青蓝系：可招聘、状态、数据强调
- 暖金/杏色：会员或特殊权益

原则：

- 强调色必须“点到为止”
- 不能把强调色铺成大面积主背景

## 10. 交互规范

### 10.1 Hover 与 Focus

- 所有 hover 效果都应轻量
- 默认是“有反馈但不跳”
- 禁止使用明显位移、发光、膨胀造成廉价感

### 10.2 固定与滚动

- 顶部栏固定
- 侧栏可固定
- 页面标题头区不固定，但滚动起点必须整齐

### 10.3 弹层

- 弹层不得被父容器裁切
- 弹层与触发器之间保持紧密但有呼吸感
- 弹层默认高于所有内容层

## 11. 内容文案规范

### 11.1 应删除的信息

后续默认不展示以下类型文案，除非它们直接影响操作：

- 当前角色的技术标识
- 单位 `none`
- session 切换成功提示
- 组织创建成功提示
- 头像旁的冗余英文身份

### 11.2 文案风格

- 面向操作，不面向介绍
- 尽量短句
- 避免“系统在自言自语”

## 12. 已确认的锁定决策

以下内容视为当前默认规范：

- 顶部栏固定高度 `68px`
- 顶部栏滚动时固定不消失
- 页面大标题头区高度 `120px`
- 标题组整体垂直居中
- 头像区只保留圆形头像入口
- 账户菜单为 hover 下拉
- 组织创建 / session 切换提示不进入主内容
- 顶部不展示 `org_admin`、`none` 等无效信息

## 13. 后续 UI 调整工作方式

从现在开始，后续 UI 修改默认按下面流程执行：

1. 先判断需求是否已被本规范覆盖
2. 若已覆盖，直接按规范实现，不再重复讨论基础细节
3. 若需求超出规范，再补充规范后实现
4. 若实现与规范冲突，以更新后的规范为准

## 14. 当前代码实现备注

当前 `apps/web/src/styles.css` 已存在大量历史叠加规则，后续应逐步清理为：

1. `tokens`
2. `layout`
3. `components`
4. `overrides`

在完成这一轮样式收敛前，后续改动必须以本规范为语义准绳，避免继续通过重复覆盖堆叠出更多不可控样式。
