## 1. 文档目标

本文档用于将《OpenClaw Team OS》PPT 中的产品愿景转化为一份可执行的客户端产品需求文档，明确以下内容：

1. 产品要解决的核心问题是什么。
2. 第一阶段应该服务谁，不服务谁。
3. MVP 必须包含哪些页面、流程与功能。
4. 哪些能力属于后续阶段，避免首版过度平台化。
5. 产品、设计、研发、运营后续如何围绕同一目标推进。

## 2. 产品背景与机会判断

### 2.1 行业背景

当前 AI 产品正在从单轮对话逐步走向执行与协作，但市场主流产品仍存在明显断层：

- 极客向产品强调 Agent 编排、Prompt、API、工作流搭建，能力强但门槛高。
- 单助手类 SaaS 使用简单，但只提供单点能力，无法支撑真实团队协作。
- 多 Agent 框架与 Runtime 已具备一定底层执行能力，但缺少面向非技术用户的组织层、治理层与结果层。

市场已经证明用户愿意为 AI 效率付费，但大量潜在用户仍然“不敢放手、不知道怎么配、不知道如何管理”。因此，下一个阶段的产品机会不是再做一个 Agent Builder，而是做一个“让普通人像管理团队一样管理 AI 劳动力”的产品。

### 2.2 核心问题

用户不是不想用 AI，而是存在三类障碍：

1. 使用门槛高：用户不理解 Prompt、参数、Agent 架构、工具调用。
2. 控制感不足：用户不知道 AI 做了什么、为什么这样做、什么时候需要介入。
3. 治理机制缺失：没有预算、权限、审批、审计等管理能力，难以进入真实业务流程。

### 2.3 机会判断

OpenClaw 当前强在底层执行能力，弱在大众化产品体验。机会不是重造执行引擎，而是在其之上补出一层“团队化产品层”，让 AI 从“工具”升级为“可治理的团队”。

## 3. 产品愿景与定位

### 3.1 产品愿景

让非技术用户像组建真实团队一样调用 AI 劳动力。

### 3.2 产品定位

OpenClaw Team OS 是一个面向非技术用户的 AI 团队操作系统。用户无需配置 Agent，只需选择岗位或团队模板，即可雇佣一支 AI 团队，并在一个统一客户端内完成任务派发、过程审批、预算管理、结果查看与组织治理。

### 3.3 一句话定义

一个让老板、团队负责人和业务人员像管理真实员工一样管理 AI 团队的客户端工作台。

### 3.4 与市面产品的差异

OpenClaw Team OS 的差异点不在于“单个 Agent 更聪明”，而在于以下四项能力同时成立：

1. 角色 Marketplace：岗位即插即用，而不是从零搭建。
2. 一键组队：围绕业务目标快速雇佣整支 AI 团队，而非拼装零件。
3. 治理与权限：审批、预算、角色权限、审计日志内建。
4. 非技术友好：面向岗位语言、组织架构和结果界面，而不是技术配置。

## 4. 目标用户与边界

### 4.1 核心目标用户

首阶段优先聚焦高净值小团队，尤其是以下场景：

- 2-20 人创业团队
- 一人公司或超级个体
- 内容团队
- 增长与运营团队
- 销售支持团队
- 招聘支持团队

这类用户通常具备以下特征：

- 有明确业务目标与交付需求
- 愿意为结果和效率付费
- 没有完整技术团队，或不希望把精力投入到 AI 工具配置
- 需要保留人工审批与关键节点控制权

### 4.2 次级目标用户

- 20-200 人的数字化 SMB
- 企业内部相对独立、结果导向明显的业务部门
- 有数字化习惯但尚未建立成熟 AI 协作体系的负责人

### 4.3 暂不服务用户

- 泛消费者娱乐类用户
- 强监管、私有化定制要求极高的大企业
- 低数字化、流程高度线下的传统行业
- 希望高度自由编排底层 Agent 的极客开发者

## 5. 产品形态与设计原则

### 5.1 产品形态

首版产品形态定义为桌面优先的客户端工作台，核心视角不是“聊天窗口”，而是“团队管理界面”。

建议首版默认采用以下交互形态：

- 左侧为组织与导航结构
- 中间为团队工作流与结果面板
- 右侧为审批、上下文、任务详情或日志抽屉

### 5.2 设计原则

1. 岗位语言优先：用“研究员、写手、审核员”替代“Agent A、Chain B、Tool C”。
2. 结果界面优先：用户首先看到产出、待审批项、成本和进度，而不是运行细节。
3. 人工控制优先：关键节点必须支持 Human-in-the-loop。
4. 团队视角优先：优先展示组织关系、上下级、委派和协作，而不是单 Agent 对话。
5. 渐进式复杂度：默认简单，高级能力按需展开，避免首屏吓退非技术用户。
6. 治理能力内建：预算、权限、审计不是企业版附加件，而是产品基础设施。

## 6. 首个切口与 MVP 业务范围

### 6.1 首个切口

第一切口为“雇佣一个 AI 增长/运营团队”。

建议首版默认提供一个官方团队模板：`AI 增长与内容运营组`。

### 6.2 选择该切口的原因

- 业务需求高频，用户易理解
- 工作链条标准化程度较高，适合模板化
- 人工审批容易嵌入关键节点
- ROI 可衡量，可通过产出数量、质量和节省人力来验证
- 风险低于财务、法务、医疗等高风险场景

### 6.3 首版团队组成建议

`AI 增长与内容运营组` 包含以下标准岗位：

1. 研究员：收集信息、竞品、选题与素材。
2. 内容策划：生成内容方向、选题建议、传播角度。
3. 文案写手：撰写初稿、改写多版本。
4. 审核员：检查事实、风格、合规项，形成待审批结果。
5. 分发助理：根据审批结果执行发布或生成待发布素材包。

### 6.4 MVP 目标

首版目标不是成为完整开放平台，而是验证以下命题：

“非技术用户是否愿意以雇佣团队的方式，持续使用 AI 完成可衡量的增长/运营工作，并在具备审批、预算与审计保障的前提下形成付费。”

## 7. 用户角色定义

### 7.1 外部用户角色

#### 7.1.1 组织管理员

- 通常为老板、创始人、部门负责人
- 负责开通团队、设置预算、审批规则、成员权限
- 关注 ROI、成本、效率、风险

#### 7.1.2 业务操作者

- 通常为运营、内容、增长、销售支持角色
- 负责提交任务、查看进展、处理待审批内容、使用结果
- 关注产出质量、时效和协作便利性

#### 7.1.3 审批人

- 可能与管理员重合，也可能为具体业务负责人
- 负责在关键节点批准、驳回或修改 AI 产出
- 关注风险控制、品牌一致性、内容准确性

### 7.2 系统内 AI 角色

- Manager：团队协调者，负责任务拆解与岗位分配
- Researcher：信息检索与研究
- Planner：规划与内容策略
- Writer：内容生成与版本扩写
- Reviewer：质量检查与风险筛查
- Publisher/Operator：分发执行与结果回传

说明：首版可以不对外暴露全部底层角色，只以“岗位名”呈现给用户。

## 8. 核心用户场景

### 8.1 场景 A：老板第一次开通 AI 团队

用户希望快速拥有一支可用的 AI 团队，不愿理解技术细节。

成功标准：

- 用户 10 分钟内完成团队创建
- 用户无需编写 Prompt 和工作流
- 用户能够清晰理解每个岗位做什么、怎么协作、哪些节点需要自己审批

### 8.2 场景 B：运营人员发起一个内容增长任务

用户输入目标与要求，例如“本周围绕某产品做 5 条小红书内容，风格偏专业但轻松”，系统自动调用团队分工协作，最终返回可审批、可修改、可分发的结果包。

成功标准：

- 用户通过自然业务语言即可发起任务
- 用户能看到任务经过了哪些岗位
- 用户能在关键节点审批或驳回
- 用户能直接看到最终产出与成本

### 8.3 场景 C：负责人控制预算和风险

用户希望限制日预算、月预算、单任务预算，并审计每一步 AI 调用与人工操作。

成功标准：

- 用户可设置预算上限和审批阈值
- 达到阈值时自动暂停或要求人工确认
- 用户能追溯每次任务的运行过程和关键决策

### 8.4 场景 D：团队复用成功打法

用户完成一类任务后，希望沉淀成组织模板，以便团队后续重复调用。

成功标准：

- 用户可基于官方团队模板进行轻量配置
- 同一组织内可复用已验证流程
- 后续阶段支持将模板沉淀到 Marketplace

## 9. 产品目标与成功指标

### 9.1 北极星指标

月度活跃的“已雇佣 AI 团队”数量，以及其持续完成任务的频率。

### 9.2 首版核心指标

- 团队开通转化率：从进入产品到成功雇佣首支团队的比例
- 首任务完成率：已开通团队的用户中，成功完成首个任务的比例
- 周留存：首周内再次发起任务的组织占比
- 审批完成率：待审批项中被有效处理的比例
- 单组织周任务数：反映团队使用深度
- 单组织月收入：验证付费模型
- 单任务平均成本：验证预算控制能力
- 人工节省时长：基于用户反馈与任务记录估算

### 9.3 首版定性目标

- 用户把产品描述为“我雇了一个 AI 团队”，而不是“我用了一个 AI 工具”
- 用户愿意把重复性业务持续交给系统执行
- 用户对审批、预算和审计有足够信任

## 10. 信息架构与页面规划

### 10.1 MVP 页面清单

建议 MVP 至少包含以下 8 个核心页面或一级视图：

1. 欢迎与团队开通页
2. 团队 Marketplace 页
3. 团队详情与雇佣确认页
4. 团队工作台页
5. 任务发起页
6. 审批中心页
7. 预算与审计页
8. 结果看板页

### 10.2 页面说明

#### 10.2.1 欢迎与团队开通页

目标：

- 让用户快速理解“这是一个 AI 团队工作台”
- 引导用户选择场景并开通第一支团队

关键内容：

- 产品价值说明
- 官方推荐团队模板
- 3 步开通引导
- 示例成果展示

#### 10.2.2 团队 Marketplace 页

目标：

- 展示可雇佣的岗位或团队模板
- 降低用户理解成本

关键内容：

- 官方团队包卡片
- 团队包说明、适用场景、可交付结果、预计成本
- 官方认证标识
- 预估 ROI 文案

#### 10.2.3 团队详情与雇佣确认页

目标：

- 帮助用户理解这支团队由哪些岗位组成、如何协作、风险点在哪里
- 完成雇佣前配置

关键内容：

- 团队组织图
- 岗位职责
- 默认审批点
- 默认预算额度
- 可修改项与不可修改项
- 开通按钮

#### 10.2.4 团队工作台页

目标：

- 成为用户日常使用主界面
- 汇总任务、成员、待办、成本和产出

关键内容：

- 团队组织图
- 今日进展
- 待审批事项
- 任务列表
- 最近产出
- 成本消耗概览

#### 10.2.5 任务发起页

目标：

- 用业务语言发起任务
- 尽量少暴露技术概念

关键内容：

- 任务目标输入
- 期望产出类型
- 截止时间
- 风格或品牌要求
- 预算提示
- 审批流程提示

#### 10.2.6 审批中心页

目标：

- 让用户集中处理待审批节点
- 建立可控感与安全感

关键内容：

- 待审批列表
- 每项产出预览
- 批准、驳回、要求重做
- 审批意见
- 风险提示

#### 10.2.7 预算与审计页

目标：

- 让用户看到钱花在哪里、系统做了什么

关键内容：

- 日/周/月预算
- 已消耗额度
- 预算预警
- 调用日志
- 人工操作日志
- 异常事件记录

#### 10.2.8 结果看板页

目标：

- 让用户看到团队价值，而不仅是过程

关键内容：

- 已完成任务数
- 内容产出数
- 审批通过率
- 平均交付时长
- 成本趋势
- 节省时长估算

## 11. MVP 功能需求

### 11.1 模块 A：团队模板与一键雇佣

#### 11.1.1 目标

让用户不需要搭建 Agent，就能快速获得一支可工作的 AI 团队。

#### 11.1.2 功能需求

1. 系统提供至少 1 个官方团队模板。
2. 每个团队模板包含岗位列表、职责说明、协作流程、审批点与预算建议。
3. 用户可查看模板详情后执行“雇佣”操作。
4. 雇佣时允许用户配置组织名称、团队名称、预算上限和审批人。
5. 雇佣完成后自动创建团队工作台实例。

#### 11.1.3 验收标准

- 新用户无需配置底层模型参数即可完成团队开通
- 开通过程控制在 10 分钟内
- 用户能理解岗位关系与默认流程

### 11.2 模块 B：组织架构与岗位视图

#### 11.2.1 目标

用组织图替代技术编排图，让用户理解团队结构与协作关系。

#### 11.2.2 功能需求

1. 系统以组织图形式展示 Manager 与各岗位关系。
2. 支持查看每个岗位的职责、输入、输出和当前状态。
3. 支持展示上下游关系，例如“研究员 -> 写手 -> 审核员 -> 分发助理”。
4. 首版支持只读组织图，不要求支持复杂拖拽编排。

#### 11.2.3 验收标准

- 用户在不学习技术概念的情况下理解协作逻辑
- 组织图能够用于查看当前团队配置与任务流向

### 11.3 模块 C：任务发起与委派

#### 11.3.1 目标

让用户通过自然业务语言下达任务，并由系统自动分解与委派。

#### 11.3.2 功能需求

1. 用户可输入任务目标、背景、限制条件、交付形式与截止时间。
2. 系统自动将任务分派给对应岗位。
3. 用户可查看任务当前所处节点与负责人岗位。
4. 任务支持状态流转：待开始、执行中、待审批、已完成、已驳回、已暂停。
5. 用户可重新执行某个任务节点。

#### 11.3.3 验收标准

- 用户可用业务语言完成任务创建
- 系统能展示清晰任务状态
- 用户能定位卡点和责任岗位

### 11.4 模块 D：审批流与 Human-in-the-loop

#### 11.4.1 目标

在关键节点保留人工控制权，建立信任。

#### 11.4.2 功能需求

1. 系统支持预设审批节点。
2. 审批动作至少包括批准、驳回、要求修改。
3. 审批时支持填写意见。
4. 系统可根据任务类型或预算阈值自动触发审批。
5. 未经审批的关键内容不得自动进入下游执行。

#### 11.4.3 验收标准

- 用户能够清楚知道哪些任务正在等待人工介入
- 所有关键节点均可审计
- 驳回后系统能够回退或重新生成

### 11.5 模块 E：预算控制与成本治理

#### 11.5.1 目标

避免用户因 AI 调用产生失控成本，并增强商业可付费感知。

#### 11.5.2 功能需求

1. 支持组织级、团队级预算设置。
2. 支持日预算、月预算、单任务预算阈值。
3. 成本展示面向用户时采用“金额/额度”表达，而非直接暴露 Token。
4. 当预算达到阈值时，系统触发提醒、审批或暂停。
5. 用户可查看任务级成本拆分。

#### 11.5.3 验收标准

- 用户能实时查看消耗情况
- 预算超限后系统能阻止继续失控执行
- 用户不需要理解 Token 才能管理成本

### 11.6 模块 F：审计日志与可追溯性

#### 11.6.1 目标

让用户能够追踪系统做了什么、谁批准了什么、出现问题时如何回溯。

#### 11.6.2 功能需求

1. 记录关键事件日志，包括任务创建、委派、审批、驳回、发布、预算触发。
2. 支持查看日志时间、执行主体、关联任务、结果状态。
3. 支持区分 AI 动作与人工动作。
4. 首版支持只读审计，不要求复杂检索与导出。

#### 11.6.3 验收标准

- 每个任务可查看关键路径日志
- 用户能够追溯异常和责任节点

### 11.7 模块 G：结果面板与价值呈现

#### 11.7.1 目标

让用户感知产品是在交付业务结果，而不是展示技术过程。

#### 11.7.2 功能需求

1. 展示近期产出与可直接使用的结果物。
2. 展示团队效率指标，如完成任务数、平均交付时长、审批通过率。
3. 展示成本与节省时间估算。
4. 支持按任务查看对应交付物。

#### 11.7.3 验收标准

- 用户能快速回答“今天完成了什么”
- 用户能看到可被复用或导出的交付成果

## 12. 首版详细用户流程

### 12.1 新用户开通流程

1. 用户进入欢迎页。
2. 用户浏览官方推荐团队模板。
3. 用户进入模板详情页查看岗位与流程。
4. 用户填写组织信息、预算、审批人。
5. 用户确认雇佣团队。
6. 系统创建团队工作台并展示首次任务引导。

### 12.2 首次任务执行流程

1. 用户输入任务目标。
2. 系统展示预计使用岗位、预估时间、预估成本。
3. 用户确认启动。
4. 系统自动执行研究、策划、写作、审核流程。
5. 在预设节点生成待审批项。
6. 用户审批通过后系统继续执行或进入最终交付。
7. 结果进入结果面板并记录成本与日志。

### 12.3 异常处理流程

1. 当预算超限时，系统暂停并提示用户处理。
2. 当审核失败时，任务回退到指定岗位重新生成。
3. 当人工驳回时，系统记录原因并重新排队执行。

## 13. MVP 范围定义

### 13.1 P0：首版必须做

- 官方单一团队模板：AI 增长与内容运营组
- 团队 Marketplace 浏览
- 模板详情与一键雇佣
- 团队组织图展示
- 任务发起与自动流转
- 审批中心
- 预算上限与提醒
- 审计日志
- 结果看板

### 13.2 P1：首版尽量做，但可根据资源延后

- 团队模板轻量自定义
- 多团队并行管理
- 任务节点重跑
- 结果导出
- 基础品牌风格配置

### 13.3 P2：明确不在首版范围内

- 开放式 Agent Builder
- 用户自由拖拽设计复杂工作流
- 面向所有行业的 Marketplace
- 第三方创作者角色上架系统
- 企业级私有化交付
- 强监管行业模板
- 完整移动端体验

## 14. 非功能需求

### 14.1 可用性

- 新用户应在 10 分钟内完成首支团队开通
- 核心任务发起流程应在 3 分钟内完成输入与确认
- 审批操作应尽量在 2 步内完成

### 14.2 可靠性

- 核心任务状态不可丢失
- 预算、审批、日志等关键治理数据必须持久化
- 任务失败应可见、可回退、可重试

### 14.3 透明性

- 用户必须知道任务当前状态
- 用户必须知道关键节点是否需要自己审批
- 用户必须知道已发生的预算消耗

### 14.4 安全性

- 组织间数据隔离
- 权限控制基础可用
- 审批与预算设置仅管理员可修改

### 14.5 性能

- 页面首屏反馈及时
- 长任务必须具备明确进度和状态提示
- 审批与预算变更应实时生效

## 15. 数据对象建议

为了方便后续技术设计，首版建议至少抽象以下核心实体：

- Organization：组织
- User：用户
- TeamTemplate：团队模板
- TeamInstance：已雇佣团队实例
- RoleTemplate：岗位模板
- Task：任务
- TaskStep：任务节点
- ApprovalItem：审批项
- BudgetPolicy：预算策略
- CostRecord：成本记录
- AuditLog：审计日志
- Deliverable：交付物

## 16. 商业模式要求

### 16.1 收费原则

产品对用户售卖的是“团队能力与结果”，而不是底层 Token 消耗。

### 16.2 首版商业化建议

1. 基础订阅：按月订阅，包含团队额度。
2. 超额计费：超过套餐额度后按量收费。
3. 高级能力增购：高级审批规则、风险控制模块、官方高级团队模板。

### 16.3 首版价格展示原则

- 前台不以 Token 为核心计费单位
- 默认展示为“本月额度”“本任务预计花费”“当前套餐包含量”
- 用可理解的业务价值描述替代底层成本说明

## 17. 路线图建议

### 17.1 0-2 个月：探索期

- 完成 20-30 个深度用户访谈
- 确定首个团队包边界
- 绘制核心流程原型
- 用 Concierge 方式验证真实交付链条

### 17.2 2-4 个月：验证期

- 锁定 5-10 个 Design Partners
- 交付 Concierge MVP
- 收集用户对审批、预算、结果面板的真实反馈
- 验证是否存在周度复用行为与付费意愿

### 17.3 4-8 个月：产品化

- 模板、审批、预算、看板产品化上线
- 优化组织图与任务追踪体验
- 建立首版计费体系

### 17.4 8-12 个月：生态起步

- 推出 Verified Marketplace
- 上线第二个垂直团队包
- 开始积累模板、岗位和治理规则的生态护城河

## 18. 风险与应对

### 18.1 风险：产品做成另一个 Agent Builder

应对：

- 首版不开放复杂流程搭建
- 强制采用岗位语言与团队视图
- 将“配置自由度”放到后期

### 18.2 风险：用户不信任 AI 自动执行

应对：

- 强化审批流
- 提供任务追踪与审计日志
- 对高风险步骤默认人工确认

### 18.3 风险：成本不可控

应对：

- 建立预算阈值与暂停机制
- 展示任务级成本拆分
- 默认额度式售卖，避免用户恐惧 Token

### 18.4 风险：首版范围过大

应对：

- 只做一个官方团队模板
- 只打一个高频垂直场景
- 明确暂不开放 Marketplace 生态能力

## 19. 待确认问题

以下问题建议在进入原型或技术方案前进一步确认：

1. 首版客户端是 Web 优先、桌面壳优先，还是原生客户端优先。
2. 团队工作台是否以“组织图”为主视图，还是“任务列表 + 组织图切换”。
3. 首版审批粒度是按任务、按节点，还是支持混合策略。
4. 成本展示是否以“人民币金额”直出，还是以“额度点数”包装。
5. 首版交付物是否需要直接连接发布渠道，还是先停留在“待发布素材包”。
6. 用户是否需要邀请真人成员协同处理审批与复核。
7. 与 OpenClaw Runtime 的边界如何划分，哪些能力由底层提供，哪些在产品层实现。

## 20. 结论

OpenClaw Team OS 首版不应被定义为“一个更强的 Agent 工具”，而应被定义为“一个让非技术用户雇佣和管理 AI 团队的客户端产品”。

首版最关键的产品策略是：

1. 不做平台空壳，先做一个真正可用的官方团队包。
2. 不卖技术配置，卖团队能力、控制感和业务结果。
3. 不急着开放生态，先把审批、预算、审计和模板沉淀做扎实。

一句话收束：

先把一个 AI 团队做成，再把一类团队做成，最后再把平台做成。

## 附录 A. 客户端专项要求

由于本产品被定义为“客户端工作台”，而非单纯网页表单系统，首版在交互和运行层面建议补充以下专项要求。

### A.1 长任务可感知

AI 团队执行的任务往往不是秒级完成，客户端必须具备长任务可感知能力：

- 在任务执行中持续显示状态、阶段、当前岗位与预计下一步动作
- 当任务进入待审批、失败、超预算等关键状态时提供明显提示
- 用户关闭当前页面后，重新进入产品仍可恢复任务上下文

### A.2 通知机制

客户端应支持关键事件通知，优先覆盖以下场景：

- 有新的待审批项
- 任务执行完成
- 任务执行失败
- 达到预算阈值
- 分发结果可查看

首版可以采用应用内通知中心加系统通知的组合方式。

### A.3 工作台连续性

客户端应具备“持续办公”感，而不是每次重新从零进入单个任务：

- 保留最近查看的团队与任务状态
- 支持从通知直接跳转到对应审批项或任务详情
- 支持从团队工作台继续发起相似任务

### A.4 任务详情抽屉

对于长链路任务，建议采用右侧详情抽屉承载以下信息：

- 任务背景
- 当前节点
- 已产出内容预览
- 审批记录
- 成本记录
- 审计日志摘要

这样可以在不离开工作台主视图的前提下处理大部分操作。

### A.5 客户端首版验收补充

- 用户在工作台上能够同时感知团队状态、待办事项和最近产出
- 任务进入关键状态时，用户无需反复刷新页面即可收到提醒
- 用户中断后再次进入客户端，能够快速恢复到上次工作上下文

## 附录 B. 基础权限矩阵

首版建议采用最小可用权限模型，至少区分三类真人角色：

- 组织管理员
- 业务操作者
- 审批人

建议权限矩阵如下：

| 权限项 | 组织管理员 | 业务操作者 | 审批人 |
| --- | --- | --- | --- |
| 查看团队模板 | 是 | 是 | 是 |
| 雇佣团队 | 是 | 否 | 否 |
| 编辑组织预算 | 是 | 否 | 否 |
| 设置审批规则 | 是 | 否 | 否 |
| 发起任务 | 是 | 是 | 可选 |
| 查看任务进展 | 是 | 是 | 是 |
| 审批关键节点 | 是 | 否 | 是 |
| 驳回并要求重做 | 是 | 否 | 是 |
| 查看结果交付物 | 是 | 是 | 是 |
| 查看审计日志 | 是 | 部分 | 部分 |
| 管理团队成员 | 是 | 否 | 否 |

补充说明：

- `部分` 表示仅能看到与本人相关或与所属团队相关的日志。
- 首版不建议做过细粒度 RBAC，避免管理复杂度过高。
- 如果首版资源有限，可将“审批人”作为管理员的一种能力子集实现。
