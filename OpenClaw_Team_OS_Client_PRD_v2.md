# OpenClaw Team OS 客户端需求文档 v2

- 文档版本：v0.2
- 文档状态：Client Reset Draft
- 更新日期：2026-04-09
- 适用范围：桌面客户端 IA、交互模型、首版功能工作区
- 本文档作用：替代旧版“流程驱动工作台”客户端设计思路

## 1. 为什么要重置客户端

上一版客户端虽然已经打通了组织、团队、任务、审批、交付物的技术链路，但产品形态仍然更像：

- 一个被放进 Electron 的网站
- 一个 onboarding 流程很多的后台
- 一个以“先接入组织，再建团队，再发任务”为中心的线性流程

这和桌面客户端产品应有的体验不一致。

用户真正想要的不是“先理解系统流程”，而是：

1. 打开客户端，先看到明确的功能入口。
2. 点击左侧图标，直接进入某个功能工作区。
3. 在某个功能工作区里直接完成操作。
4. 把 Bot 员工、Bot 团队、任务和审批看成软件里的对象，而不是教程里的步骤。

因此，客户端需要从“流程驱动”重置为“功能驱动”。

## 2. 新的产品定义

### 2.1 一句话定义

OpenClaw Team OS 客户端是一个“AI 员工与 AI 团队雇佣工作台”。

### 2.2 客户端的第一原则

客户端不是聊天页，也不是网站首页，而是一个桌面工作台：

- 左侧是一列图标，代表不同功能
- 右侧是当前功能对应的完整工作区
- 用户的主要行为是切换功能、雇佣员工/团队、派发任务、处理审批

### 2.3 客户端的首版主心智

首版客户端主心智不是“先学会 Team OS”，而是：

`先招聘一个 Bot 员工或一支 Bot 团队，然后立即开始使用。`

## 3. 新的信息架构

## 3.1 总体结构

客户端分为两层：

1. 全局导航层
2. 功能工作区层

### 3.1.1 全局导航层

位于客户端左侧，固定为窄栏图标导航，不承载复杂信息，只负责功能切换。

建议首版图标顺序如下：

1. 总览
2. 招聘
3. 员工
4. 团队
5. 任务台
6. 审批
7. 设置

### 3.1.2 功能工作区层

位于客户端右侧，占据主要空间，每个功能都有自己的工作区布局与交互逻辑。

## 3.2 功能说明

### 3.2.1 总览

作用：快速理解当前组织的 AI 编制状态。

展示内容：

- 已招聘员工数量
- 已招聘团队数量
- 待审批数量
- 本月消耗
- 推荐招聘模块
- 当前激活的单位

### 3.2.2 招聘

作用：客户端首版的核心工作区。

展示内容必须分成两大区块：

1. 招聘 Bot 员工
2. 招聘 Bot 团队

交互目标：

- 可以先按方向分类筛选
- 可以在方向下继续按细分目标筛选
- 可以直接招聘某一类单体员工
- 可以直接招聘整支团队
- 不需要先跳去复杂的组织设置页
- 方向卡片需要有明显不同的图标识别，不能只靠文字区分

### 3.2.3 员工

作用：查看已经入职的单体 Bot 员工。

展示内容：

- 左侧员工目录
- 右侧员工档案
- 员工名称，可直接改名
- 当前状态
- 核心技能
- 职责说明
- 可处理任务类型
- 快速派发任务入口

交互要求：

- 员工页必须采用“左侧名单 + 右侧详情”的客户端结构
- 不允许为了填满页面而堆大卡片
- 右侧详情区必须支持直接改名，并查看技能、职责和交付能力

### 3.2.4 团队

作用：查看已经入职的 Bot 团队。

展示内容：

- 左侧团队目录
- 右侧团队档案
- 团队名称，可直接改名
- 团队构成
- 当前状态
- 核心技能
- 职责说明
- 可执行任务方向
- 快速进入任务台

交互要求：

- 团队页沿用员工页的目录式结构
- 不能再出现大面积空白和“为了显得有内容而拉长卡片”的问题

### 3.2.5 任务台

作用：给当前员工或团队直接分配任务。

展示内容：

- 当前选中的执行单位
- 一句话目标输入
- AI 推荐执行单位
- 快捷任务模板
- 一键生成 Brief
- 自动补全 Brief
- 文本任务输入区
- 最近任务
- 最近任务快捷动作
- 最近交付物
- 待审批草稿预览
- 结果详情检视区

### 3.2.6 审批

作用：集中处理需要人工确认的任务节点。

展示内容：

- 待审批项列表
- 审批摘要
- 审批前摘要卡
- 批准 / 驳回操作

### 3.2.7 设置

作用：承载低频但必要的管理能力。

展示内容：

- 当前组织信息
- Session 信息
- 预算策略
- 邀请成员

说明：设置必须放到低频区，不能再作为首版客户端的主入口。

## 4. 首版招聘工作区定义

## 4.1 产品目标

招聘工作区必须让用户一眼理解：

- 我能招聘什么
- 招来之后能做什么
- 是单个员工还是整支团队
- 招聘动作如何发生

## 4.2 模块划分

首版招聘工作区至少包含以下模块：

### 4.2.0 方向分类模块

招聘页需要在员工模块上方提供方向分类，至少包含：

1. 游戏创作
2. 应用设计
3. AI短剧制作

分类目标：

- 让用户先按业务方向理解“该招哪类 Bot”
- 同时筛选员工与团队目录
- 避免招聘页一次性铺开全部岗位，导致认知负担过大
- 每个方向卡片必须提供图标化识别，帮助用户更快扫视

### 4.2.0.1 二级筛选模块

在方向分类下，需要增加一层细分筛选，帮助用户进一步缩小招聘范围。

建议首版至少包含：

1. 原型开发
2. 视觉设计
3. 内容增长
4. 短剧脚本
5. 整包交付

交互目标：

- 先按行业方向理解“在哪个业务面里找 Bot”
- 再按具体目标理解“这次我要招来解决什么问题”
- 同时作用于员工和团队目录

### 4.2.1 Bot 员工模块

建议首版提供：

1. 设计师员工
2. 游戏研发员工
3. 3D 美术员工
4. 运营文案员工
5. 短剧编导员工

每个员工卡片包含：

- 名称
- 能力方向
- 典型产出
- 响应时长 / 成本
- 招聘按钮

补充要求：

- 招聘按钮文案要短，优先用“招聘”而不是过长按钮
- 员工卡片的信息顺序应优先突出角色、图标和能力标签，再展示说明文字

### 4.2.2 Bot 团队模块

建议首版提供：

1. 游戏制作团队
2. 内容增长团队
3. 创意素材团队
4. AI 短剧制作团队

每个团队卡片包含：

- 团队名称
- 团队说明
- 适合处理的问题
- 预估成本
- 招聘按钮

## 4.3 招聘后的反馈

招聘成功后，界面需要立即给出三层反馈：

1. 招聘成功提示
2. 该员工或团队进入“员工 / 团队”工作区
3. 可直接跳转到“任务台”为其分配第一个任务

补充要求：

- 任务提交后的反馈不能只依赖全局 banner
- 在没有正式交付物前，任务台右侧应显示待审批草稿预览
- 任务输入区中的推荐文案只能在“切换单位”或“主动点击使用推荐任务”时回填，不能在用户删除文本时自动重置
- 任务台需要支持“一键生成 Brief”，让用户不用每次从空白文本框开始
- 任务台需要支持“一句话目标 -> AI 推荐执行单位 -> 自动补全 Brief”的低摩擦路径
- 最近任务卡片需要提供快捷动作，至少包含“去审批 / 看交付 / 复用任务”中的核心入口
- 结果区不能只是文本列表，需要提供“结果列表 + 结果详情检视”结构
- 审批页需要把长摘要压成一眼能判断的摘要卡，降低人工审批负担
- 对于 AI 短剧制作任务，结果区需要支持“剧情包 + 分镜 + 视频接力包”的专用视图
- 对于 AI 短剧制作任务，客户端需要先输出可直接投喂的视频提示词包，再逐步对接真实视频生成 API

## 4.4 AI 短剧制作专项工作流

首版客户端中，`短剧编导员工` 与 `AI 短剧制作团队` 需要支持一个更具体的交付结构：

1. 剧情钩子
2. 30 秒三幕结构
3. 关键对白
4. 分镜清单
5. 视频生成接力包

说明：

- 当前版本先以 `可直接投喂的视频接力包` 为目标，而不是假装已经完成一键出片。
- 接力包默认面向 `Seedance 2.0` 这类视频生成工具，包含主提示词、镜头提示词、画幅、时长和负面提示词。
- 当真实视频 API 接入后，再将“复制接力包”升级为“直接提交生成”。

## 5. 交互原则

### 5.1 必须取消的旧体验

以下体验在新版本中不应再作为主路径：

- 首屏大段 onboarding 文案
- 先做组织设置再开始使用
- 以页面跳转为主的流程式 UI
- 过多的解释性卡片和教程型布局
- 进入客户端首屏把无用描述放在视觉中心

### 5.2 必须强化的新体验

1. 功能切换感：左侧图标切换时，右侧工作区明确变化。
2. 软件对象感：员工、团队、任务、审批都像“系统对象”而不是“文章段落”。
3. 低摩擦操作感：招聘、切换、派发任务都应是一屏内完成。
4. 桌面客户端感：界面密度、导航方式、区域布局都应偏软件工作台，而不是营销网站。
5. 对象可管理感：已招聘员工和团队必须支持改名、查看技能和职责。

## 6. 客户端视觉与布局原则

### 6.1 布局原则

- 左侧窄栏图标导航固定
- 右侧内容区使用工作台式布局
- 不再使用大 Hero 作为登录后的主视觉
- 不再把大面积信息包装成“网站首页版头”

### 6.2 视觉原则

- 强化层级，不强化装饰
- 导航区域克制、稳定、可扫描
- 内容区以功能模块和对象列表为主
- 保持桌面软件的清晰度与操作感

## 7. 与现有技术能力的映射

本次客户端重构不要求同步推翻现有后端能力。

现阶段的技术策略如下：

1. 组织、Session、审批、预算能力继续复用当前 API。
2. 招聘动作先复用现有 `createTeamInstance` 能力做存储承接。
3. 招聘工作区中的“员工 / 团队目录”允许先由客户端定义为产品目录。
4. 后续再逐步将这些目录映射成真实的多模板后端结构。

换句话说：

`先把客户端产品模型做对，再逐步把后端模板体系跟上。`

## 8. 本轮重构的具体目标

本轮客户端优化的最低交付标准：

1. 新增一份客户端 PRD v2，明确新的 IA 和交互模型。
2. 将客户端主界面改为“左侧图标导航 + 右侧功能工作区”。
3. 将“招聘”作为第一主工作区。
4. 在招聘工作区中提供“招聘单体 Bot 员工”与“招聘 Bot 团队”两类入口。
5. 保留员工、团队、任务台、审批、设置等基础工作区骨架。
6. 让用户可以从招聘直接进入使用，而不是陷入复杂 onboarding。

## 9. 暂不在本轮解决的事情

为了保证重构聚焦，本轮先不解决以下问题：

- 多行业完整模板体系
- 手机端交互重构
- 正式商业化账号体系
- 完整的拖拽式组织架构编辑
- 高级分析报表

## 10. 当前结论

客户端接下来的核心目标不是继续加页面，而是：

`让 OpenClaw Team OS 看起来、用起来都像一个真正的 AI 劳动力客户端。`

第一步就是从“招聘一个 Bot 员工 / 团队”开始。
